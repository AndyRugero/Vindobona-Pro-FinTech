1. Einleitung

1.1. Ausgangssituation

Im heutigen Finanzsektor sind Schnelligkeit, Sicherheit und Barrierefreiheit im digitalen Banking essenzielle Standarderwartungen. Klassische Online-Banking-Plattformen wirken jedoch häufig überladen, bieten wenig Unterstützung bei Budget-Entscheidungen und verzichten meist auf integrierte KI-Beratungen oder kostenfreie, regionale Filialfinder. Das Semesterprojekt Vindobona Pro FinTech setzt genau an diesem Punkt an. Es schafft ein modernes, visuell ansprechendes Web-Banking-Dashboard, das speziell für den Großraum Wien konzipiert wurde.

1.2. Zielsetzung

Das Ziel dieses Projekts ist die Konzeption und Realisierung einer voll funktionsfähigen FinTech-Webanwendung. Diese verbindet typische Kernbanken-Features (Kontoführung, Ledger, Währungs-Wallets und P2P-Überweisungen) mit intelligenten Mehrwertdiensten wie einem KI-Finanzberater (Chatbot) und einer interaktiven Geldautomaten- und Filialsuche auf Basis freier Kartendaten (Leaflet). Das System wird nach modernsten DevOps-Prinzipien (Docker, Kubernetes) auf Cloud-Diensten (Azure, Vercel) ausfallsicher betrieben.

1.3. Zielgruppe

  • Privatkunden (Endbenutzer): Verwalten ihr Guthaben in verschiedenen Währungen, buchen Transaktionen, nutzen den KI-Assistenten zur Analyse ihrer Ausgaben und lokalisieren Wiener Geldautomaten.
  • Administratoren (Bank-Mitarbeiter): Überwachen Kontostände, sperren/entsperren Kreditkarten bei Sicherheitsvorfällen und verwalten Benutzerrollen.
  • Auditoren (Compliance-Beauftragte): Analysieren Systemereignisse über einen lückenlosen, scrollbaren Audit-Log-Feed mit IP-Adressen und Zeitstempeln.

2. Projektbeschreibung

2.1. Vision

Vindobona Pro FinTech verbindet erstklassige Ästhetik (Dark Mode, Glassmorphismus, BEM-CSS-Richtlinien) mit professioneller Performance und Skalierbarkeit. Die Anwendung bietet dem Nutzer das Gefühl eines modernen Neo-Brokers, während im Hintergrund robuste Authentifizierung (JWT, Google OAuth, 2FA) und ausfallsichere Kubernetes-Architekturen mit Self-Healing-Fähigkeiten (Liveness/Readiness Probes) für maximale Zuverlässigkeit sorgen.

2.2. Hauptmerkmale (Scope)

Die Anwendung implementiert die folgenden funktionalen Kernkomponenten:

  • Benutzerkonto & Authentifizierung: Sichere Registrierung und Anmeldung via JSON Web Token (JWT), Multi-Faktor-Authentifizierung (2FA), Passwort-Reset per E-Mail und Google OAuth Integration.
  • Multi-Währungs-Wallets: Parallele Kontoführung in verschiedenen Leitwährungen (EUR, USD, GBP) mit Echtzeit-Tausch und Saldenanzeige.
  • Echtzeit-Ledger (Hauptbuch): Chronologische Auflistung aller Transaktionen inklusive automatisierter, regelbasierter Kategorisierung (z. B. "Apple" zu "Tech", "Netflix" zu "Entertainment").
  • AI Chatbot Assistent: Dialog-Interface zur Auswertung persönlicher Finanzdaten und intelligenter Beratung ("Wie viel habe ich diesen Monat für Essen ausgegeben?").
  • Wiener ATM- & Filialfinder: Kostenlose, interaktive OpenStreetMap-Karte mit benutzerdefinierten Markern für alle Bankstandorte in Wien.
  • Budget-Manager: Erstellung monatlicher Ausgaben-Limits pro Kategorie und Echtzeit-Fortschrittsbalken mit Farbindikator (Rot bei Überschreitung).
  • Mitglieder-Überweisungen: Direktüberweisungen an andere registrierte Systemmitglieder über ein sicheres Buchungspanel.
  • Admin-Konsole & Live-Audit: Exklusiver Bereich für Admins zum Sperren von Karten und Verfolgen sicherheitsrelevanter Aktionen in Echtzeit.
  • Kontoauszug-Export: Downloadmöglichkeit des Ledgers im CSV- und PDF-Format für persönliche Aufzeichnungen.

3.2. Nicht-funktionale Anforderungen (NFA)

  • NFA-100: Sicherheit & DSGVO-Konformität
    • Passwortspeicherung erfolgt ausschließlich gehasht mit dem bcrypt-Algorithmus.
    • Sitzungs-Tokens (JWT) enthalten keine sensitiven Daten und besitzen eine Gültigkeit von maximal 24 Stunden.
    • Lückenlose Isolierung der Produktionsdatenbank über Azure-Firewall-Richtlinien.
  • NFA-200: Usability & Frontend-Design
    • Die Applikation nutzt ein einheitliches Dark-Mode-Design mit CSS Glassmorphismus.
    • Die Navigation ist vollständig barrierefrei und responsiv (optimiert für Smartphones, Tablets und Desktops).
    • Einsatz von Micro-Animations bei UI-Aktionen (z. B. Hover-Effekt bei StatCards).
  • NFA-300: Performance & Robustheit
    • Die API-Antwortzeiten liegen unter Normalbedingungen unter 500 ms.
    • Implementierung eines Rate-Limiters auf der API-Ebene (max. 100 API-Anfragen pro IP in 15 Minuten).
  • NFA-400: Deployment & Infrastruktur
    • Das Backend läuft in Docker-Containern und wird in einem Kubernetes-Cluster mit mindestens 3 Replika-Pods gehostet (Ausfallsicherung).
    • Die Pipeline-Automatisierung (CI/CD) erfolgt über GitHub Actions.

4. Architektur und Technologie

4.1. Systemarchitektur

Das Gesamtsystem gliedert sich in ein clientseitiges React-Frontend und eine serverseitige Node.js Express REST API mit angebundenen SQL-Datenbanken.

4.2. Technologieschnitt (Tech Stack)

  • Frontend: React 19.x, TypeScript (TSX), Vite, Vanilla CSS (BEM-Konvention), Lucide Icons, Leaflet Maps.
  • Backend: Node.js, Express.js, JWT, `express-rate-limit`.
  • Datenbanken: SQLite (für die lokale Entwicklung), PostgreSQL Flexible Server auf Microsoft Azure (für die Cloud-Produktion).
  • Infrastruktur & DevOps: Docker, Kubernetes (YAML manifests), GitHub Actions (CI/CD), Vercel (Frontend-Hosting), Azure Web App for Containers (Backend-Hosting).

9. Mögliche Anhänge

Anhang A: Datenbankschema (DDL)

Nachfolgend ist das strukturierte Schema der relationalen SQL-Datenbank zur Speicherung der Kernentitäten dargestellt:

CREATE TABLE IF NOT EXISTS users (
    id TEXT PRIMARY KEY,
    username TEXT UNIQUE NOT NULL,
    email TEXT UNIQUE,
    password_hash TEXT NOT NULL,
    google_id TEXT UNIQUE,
    is_verified INTEGER NOT NULL DEFAULT 0,
    two_factor_secret TEXT,
    two_factor_enabled INTEGER NOT NULL DEFAULT 0,
    balance REAL NOT NULL DEFAULT 0.0,
    role TEXT NOT NULL DEFAULT 'user',
    is_card_frozen INTEGER NOT NULL DEFAULT 0,
    avatar_url TEXT
);

CREATE TABLE IF NOT EXISTS transactions (
    id TEXT PRIMARY KEY,
    date TEXT NOT NULL,
    receiver TEXT NOT NULL,
    amount REAL NOT NULL,
    category TEXT DEFAULT 'General',
    is_negative INTEGER NOT NULL,
    status TEXT DEFAULT 'Complete',
    user_id TEXT,
    FOREIGN KEY(user_id) REFERENCES users(id)
);

CREATE TABLE IF NOT EXISTS wallets (
    id TEXT PRIMARY KEY,
    user_id TEXT NOT NULL,
    currency TEXT NOT NULL,
    balance REAL NOT NULL DEFAULT 0.0,
    FOREIGN KEY(user_id) REFERENCES users(id),
    UNIQUE(user_id, currency)
);

CREATE TABLE IF NOT EXISTS budgets (
    id TEXT PRIMARY KEY,
    user_id TEXT,
    category TEXT NOT NULL,
    amount REAL NOT NULL,
    FOREIGN KEY(user_id) REFERENCES users(id),
    UNIQUE(user_id, category)
);

CREATE TABLE IF NOT EXISTS audit_logs (
    id TEXT PRIMARY KEY,
    user_id TEXT,
    action TEXT NOT NULL,
    details TEXT,
    ip_address TEXT,
    timestamp TEXT NOT NULL,
    FOREIGN KEY(user_id) REFERENCES users(id)
);

Anhang B: API-Schnittstellendokumentation

  • POST /api/users/register - Registriert einen neuen Account. Body: { username, email, password }.
  • POST /api/auth/login - Überprüft Anmeldedaten und gibt JWT-Token zurück. Body: { username, password }.
  • GET /api/transactions - Ruft alle Transaktionen des angemeldeten Benutzers ab (erfordert JWT-Header).
  • POST /api/transactions - Bucht eine neue Transaktion. Body: { date, receiver, amount, category, isNegative }.
  • POST /api/transactions/transfer - Führt eine P2P-Überweisung an ein anderes Mitglied aus. Body: { receiverEmail, amount }.
  • GET /api/budgets - Ruft gesetzte Budgetlimits ab.
  • POST /api/budgets - Speichert ein neues Budgetlimit. Body: { category, amount }.
  • GET /api/locations - Ruft die Koordinaten der Wiener Geldautomaten und Filialen ab (Leaflet-Anbindung).
  • POST /api/chat - Sendet Fragen an den LLM-Finanzassistenten. Body: { message }.
  • GET /api/admin/logs - Ruft die Audit-Logs zur Live-Überwachung ab (erfordert Administrator-JWT-Token).
